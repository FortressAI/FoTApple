Tests

Test Types: - Unit Tests: Test individual functions/classes in isolation - Integration Tests: Test component interactions - Validation Tests: Domain-specific correctness (medical accuracy, legal rules, etc.)

Writing Tests:

import XCTest
@testable import FoTClinician

final class MedicationServiceTests: XCTestCase {
    var service: MedicationService!

    override func setUp() async throws {
        service = MedicationService()
    }

    func testDrugInteractionDetection() async throws {
        // Given: Patient on warfarin
        let patient = Patient(
            medications: [
                Medication(name: "Warfarin", dose: "5mg", frequency: "daily")
            ]
        )

        // When: Checking aspirin interaction
        let result = try await service.checkInteraction(
            patient: patient,
            newDrug: "Aspirin"
        )

        // Then: Major interaction detected
        XCTAssertEqual(result.severity, .major)
        XCTAssertTrue(result.description.contains("bleeding risk"))
        XCTAssertNotNil(result.recommendation)
    }
}

Test Requirements: - ✅ All new code has tests - ✅ Tests are deterministic (no flaky tests) - ✅ Tests run quickly (< 1 second each) - ✅ Tests are isolated (no dependencies between tests) - ✅ Edge cases covered - ✅ Error cases tested

Pull Request Process

1. Before You Start

☐ Create Issue or Discussion - Describe what you plan to work on - Get feedback from maintainers - Ensure work aligns with project goals

☐ Fork Repository

# On GitHub, click "Fork" button
# Then clone your fork
git clone https://github.com/YOUR_USERNAME/FoTApple.git
cd FoTApple

# Add upstream remote
git remote add upstream https://github.com/FortressAI/FoTApple.git

2. Create Branch

Branch Naming: - fix/issue-number-description (bug fixes) - feature/feature-name (new features) - docs/what-youre-documenting (documentation) - test/what-youre-testing (tests only) - refactor/what-youre-refactoring (code improvements)

Example:

git checkout -b feature/add-prescription-validation

3. Make Changes

☐ Write Code - Follow Code Style Guidelines - Keep changes focused (one feature/fix per PR) - Break large changes into smaller PRs

☐ Add Tests - Test all new functionality - Add regression tests for bug fixes - Run swift test to verify

☐ Update Documentation - Update code comments - Update wiki pages if user-facing changes - Update README if architecture changes

☐ Commit - Write clear, descriptive commit messages - Follow Commit Message Guidelines

4. Test Your Changes

☐ Run All Tests

swift test

☐ Run Specific Tests

cd packages/FoTClinician
swift test

☐ Check Code Coverage

swift test --enable-code-coverage

☐ Manual Testing - Build and run iOS app - Test on actual device (if possible) - Verify UI changes look good

5. Sync with Upstream

☐ Fetch Latest Changes

git fetch upstream
git rebase upstream/main

☐ Resolve Conflicts (if any)

# Fix conflicts in files
git add .
git rebase --continue

6. Push and Create PR

☐ Push to Your Fork

git push origin feature/add-prescription-validation

☐ Create Pull Request on GitHub 1. Go to github.com/FortressAI/FoTApple 2. Click "New Pull Request" 3. Select your branch 4. Fill out PR template (see below) 5. Click "Create Pull Request"

8. Code Review

What to Expect: - Maintainers will review your PR (usually within 1-3 business days) - You may be asked to make changes - Discussion is encouraged - ask questions! - Be patient and respectful

Responding to Feedback:

# ... edit code ...

9. CI/CD Checks

Required Checks: - ✅ All tests pass - ✅ Code compiles without errors - ✅ SwiftLint passes (with reasonable exceptions) - ✅ Code coverage ≥ 80% for new code - ✅ No security vulnerabilities detected

If Checks Fail: 1. Click on failed check to see details 2. Fix the issue locally 3. Push updated code 4. Checks run automatically

10. Merge

When Your PR is Approved: - Maintainer will merge PR - Your changes appear in main branch - You're credited as contributor! 🎉

After Merge:

# Update your local repository
git checkout main
git pull upstream main

# Delete feature branch
git branch -d feature/add-prescription-validation
git push origin --delete feature/add-prescription-validation

Code Style Guidelines

Swift Style

Follow: - Swift API Design Guidelines - SwiftLint Rules

Key Principles: - Clarity: Code is read more than written - Consistency: Follow existing patterns - Simplicity: Favor simple solutions over clever ones

Naming:

// ✅ Good: Clear, descriptive names
func calculateBodyMassIndex(weight: Double, height: Double) -> Double

// ❌ Bad: Abbreviations, unclear purpose
func calcBMI(w: Double, h: Double) -> Double

Formatting:

// ✅ Good: Consistent indentation, spacing
func checkDrugInteraction(
    drug1: String,
    drug2: String
) async throws -> InteractionResult {
    let result = try await rxNavClient.checkInteraction(
        drug1: drug1,
        drug2: drug2
    )
    return result
}

// ❌ Bad: Inconsistent spacing
func checkDrugInteraction(drug1:String,drug2:String)async throws->InteractionResult{
let result=try await rxNavClient.checkInteraction(drug1:drug1,drug2:drug2)
return result}

Comments:

// ✅ Good: Explain why, not what
// Use exponential backoff to avoid overwhelming RxNav API
let delay = baseDelay * pow(2, attemptNumber)

// ❌ Bad: States the obvious
// Multiply base delay by 2 raised to attempt number
let delay = baseDelay * pow(2, attemptNumber)

Documentation:

Commit Message Guidelines

Format

Type: Brief description (50 chars max)

Longer explanation if needed (wrap at 72 chars). Explain what and why, 
not how. Include motivation for change and contrast with previous behavior.

- Bullet points okay
- Use present tense ("Add feature" not "Added feature")
- Reference issues and PRs

Fixes #123
Closes #456

Types

  • Fix: Bug fixes
  • Add: New features
  • Update: Changes to existing features
  • Remove: Removing features or code
  • Refactor: Code restructuring without behavior change
  • Docs: Documentation only
  • Test: Adding or updating tests
  • Style: Formatting, whitespace (no code change)
  • Perf: Performance improvements
  • Security: Security-related changes

Examples

Good:

Fix: Drug search timeout after 30 seconds

Increase timeout from 10s to 30s for slow connections and add retry
logic with exponential backoff. Previously, searches would fail on
slower networks even if the API was responding.

- Increase initial timeout to 30s
- Add retry with exponential backoff (max 3 attempts)
- Add test for timeout behavior
- Add test for retry logic

Fixes #123

Good:

Add: Prescription validation for controlled substances

Implements DEA schedule validation and quantity limits for controlled
substance prescriptions per state and federal regulations.

- Add DEA schedule lookup
- Implement quantity limits by schedule
- Add state-specific restrictions
- Add comprehensive test suite

Closes #456

Bad:

Fixed some bugs

Bad:

updated code to make it work better and also fixed a typo in the
documentation and refactored some functions and added tests

Testing Guidelines

Test Coverage Requirements

  • New Code: ≥ 80% coverage
  • Modified Code: Maintain or improve coverage
  • Critical Paths: 100% coverage (e.g., drug interaction checking, FRCP deadline calculation)

Test Types

Unit Tests:

// Test single function in isolation
func testBMICalculation() {
    let bmi = calculateBMI(weight: 70, height: 1.75)
    XCTAssertEqual(bmi, 22.86, accuracy: 0.01)
}

Integration Tests:

// Test components working together
func testDrugInteractionWorkflow() async throws {
    let service = MedicationService()
    let patient = createTestPatient(medications: ["Warfarin"])
    let result = try await service.checkInteraction(
        patient: patient,
        newDrug: "Aspirin"
    )
    XCTAssertEqual(result.severity, .major)
}

Validation Tests:

// Test domain-specific correctness
func testMedicalBoardExam_Cardiology() async throws {
    let testCase = loadTestCase("cardiology-001-stemi")
    let diagnosis = try await clinician.diagnose(testCase.patientData)
    XCTAssertEqual(diagnosis.primary, testCase.expectedDiagnosis)
    XCTAssertGreaterThan(diagnosis.confidence, 0.90)
}
